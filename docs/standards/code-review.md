# Code Review 规范

**版本：** v1.0
**日期：** 2026-06-17

---

## 1. Review 流程

```
开发者提交 PR
    │
    ▼
Review 者收到通知
    │
    ▼
Review 者阅读代码变更
    │
    ├─ 功能正确性
    ├─ MISRA-C 合规
    ├─ 编码规范
    ├─ 测试覆盖
    │
    ▼
填写 Code Review Checklist
    │
    ▼
Review 结果
    ├─ Approve → 可合并
    ├─ Request Changes → 修改后重新 Review
    └─ Comment → 建议，不阻塞
```

---

## 2. Review 要求

### 2.1 按模块分类

| 模块类型 | Review 要求 | 示例模块 |
|----------|-------------|----------|
| 安全关键 | 必须 2 人 Approve | 计量、阀门控制、7个安全检测组件 |
| 普通模块 | 1 人 Approve | HMI、EventLog、UsageLog |
| 配置/工具 | 1 人 Approve | Debug、配置文件 |

### 2.2 Review 时效

| PR 规模 | 期望 Review 完成时间 |
|---------|----------------------|
| < 100 行 | 1 个工作日 |
| 100-300 行 | 2 个工作日 |
| > 300 行 | 3 个工作日 |

---

## 3. Review 检查项

### 3.1 功能正确性

- [ ] 功能是否符合设计文档要求？
- [ ] 边界条件是否处理（参数为 0、最大值、NULL）？
- [ ] 是否有明显的逻辑错误？
- [ ] 是否考虑了异常情况？

### 3.2 MISRA-C 合规

- [ ] 是否使用了裸 int/unsigned？（应使用 stdint.h）
- [ ] 是否有隐式类型转换？（应显式 cast）
- [ ] 是否有 goto 语句？（禁止）
- [ ] 是否有递归？（禁止）
- [ ] switch 是否有 default 分支？（必须）
- [ ] 指针参数若只读是否声明为 const？（规则 8.13）

### 3.3 编码规范

- [ ] 函数命名是否符合规范（模块前缀 + 动词）？
- [ ] 变量命名是否清晰？
- [ ] 函数长度是否 ≤ 60 行？
- [ ] 是否有过深的嵌套（> 4 层）？
- [ ] 文件头注释是否完整？

### 3.4 测试

- [ ] 是否有单元测试？
- [ ] 测试是否覆盖主要场景（正常、边界、异常）？
- [ ] 测试是否通过？

### 3.5 文档

- [ ] 公开 API 是否有注释说明？
- [ ] 复杂逻辑是否有注释？
- [ ] 是否需要更新设计文档？

---

## 4. Review 结果类型

| 结果 | 含义 | 后续动作 |
|------|------|----------|
| **Approve** | 代码合格，可以合并 | 开发者可合并 PR |
| **Request Changes** | 有问题需要修改 | 开发者修改后重新提交 Review |
| **Comment** | 建议但不强制 | 可选择采纳，不阻塞合并 |

---

## 5. Review 礼仪

### 5.1 审查者

- **对事不对人**：评论代码，不评论人
- **具体明确**：指出问题位置，说明为什么是问题
- **提供建议**：不只说"这不对"，要说"建议这样改..."
- **尊重时间**：及时 Review，不拖延

### 5.2 被审查者

- **不辩解**：接受建议，理性讨论
- **及时响应**：收到 Review 后尽快处理
- **感谢审查者**：Review 是额外的付出
